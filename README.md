# 状态栏基金 (Menubar Fund)

一款 macOS 状态栏基金净值/估值查看器。常驻顶栏,轮播显示自选基金的盘中估算净值与涨跌,点击图标弹出悬浮窗管理自选、查看走势详情。

> 中文 A 股市场习惯:**涨红跌绿**。

---

## 为什么有这个项目

[fishing-funds](https://github.com/1zilc/fishing-funds) 是一款优秀的开源基金状态栏 App。但在使用中发现其**「添加自选基金」功能失效**——经排查,根因是天天基金(东方财富)下线了实时估值接口 `fundgz.1234567.com.cn`(返回 404),而该 App 添加基金时必须调用此接口校验;更糟的是,其核心数据层私有子模块 `fishing-funds-enh` 已被作者从 GitHub 删除,无法基于原项目修复。

本项目**参考 fishing-funds 的产品形态与交互思路,从零重写**,数据源改用仍存活的 `pingzhongdata` 接口,并在此基础上实现了**盘中实时估值**(原项目依赖的接口已死的部分功能)。

**特别感谢** [1zilc/fishing-funds](https://github.com/1zilc/fishing-funds) 作者的产品设计与交互灵感。

## 功能

- 📊 **状态栏轮播**:顶栏标题轮播所有自选基金的盘中估算净值/涨跌(带基金简称),每 5 秒切换
- 📈 **盘中实时估值**:交易时段用前十大重仓股实时行情 × 持仓占比推算基金估算净值/涨幅(算法见下文)
- 🔍 **添加自选**:支持单只搜索(代码/名称/拼音),也支持**批量添加**(多代码用 `;` `,` 空格分隔)
- 📋 **悬浮窗列表**:点状态栏图标弹出,显示所有自选的净值/估算/涨跌,失焦自动收起
- 📉 **详情页**:净值走势折线图(近1月/3月/1年/全部)、收益率、**前十大重仓股实时涨跌表格**、基金经理
- 🔄 **后台刷新**:默认每 60 秒刷新
- 💾 **本地持久化**:自选列表与设置自动保存,重启不丢失
- 🌙 **深色模式**:自动跟随系统

## 盘中估值算法

盘中实时估值是本项目的核心功能(因为天天基金的 `fundgz` 实时估值接口已下线,需自行推算)。

### 数据来源(均已实测可用)

| 数据 | 接口 | 说明 |
|------|------|------|
| 收盘净值/走势/经理 | `fund.eastmoney.com/pingzhongdata/<code>.js` | 上一交易日单位净值作为估算基准 |
| 前十大重仓占比 | `POST fundmobapi.../FundMNInverstPosition` | 季报数据,含持仓占比 `JZBL`、截止日 |
| 持仓股实时行情 | `push2.eastmoney.com/api/qt/ulist.np` | 盘中实时价格 `f2`、涨跌 `f3` |

### 算法(加权混合模型)

```
估算涨幅% = Σ(持仓股实时涨幅% × 持仓占比%) / 100
估算净值 = 上一交易日单位净值 × (1 + 估算涨幅 / 100)
```

**模型说明**:基金 = 前十大重仓(已知波动)+ 剩余资产(未知)。已知的重仓股按实际涨跌计入,剩余资产(小盘股+债券+现金)按「当日不动」假设(贡献 0),分母用 100(全基金),相当于把前十的波动摊到整个基金。

**为什么不用纯归一化(分母=前十占比)?** 纯归一化会假设「剩余资产也按同样比例波动」,对多数基金而言**高估**波动。实际剩余 50%+ 资产波动通常小于重仓股,混合模型更接近真实。真实值介于两者之间。

**可信度(覆盖率)**:UI 会显示 `覆盖资产 XX%`,即前十大重仓占基金总资产的比例。占比越高(如 80%),估算越接近真实;占比低(如 30%),偏差较大,仅供参考。

### 局限(诚实说明)

- **持仓数据滞后**:占比来自季报(有 1-3 个月延迟),基金经理调仓后估算会失真。UI 标注「持仓截止 YYYY-MM-DD」。
- **只覆盖前十**:前十通常占基金 40-60%,剩余资产未纳入。
- **非交易时段不估算**:收盘后/周末/节假日无实时行情,显示历史收盘净值。
- **非股票型基金**(纯债基/货基)持仓股少,估算不适用,自动回退收盘净值。

### 历史数据采集(为未来校准埋点)

App 每天自动采集 `{盘中估算涨幅, 收盘实际涨幅}` 配对样本,本地存储 60 天。目前仅采集不参与计算——待数据累积充分后,可用于量化估算误差、做数据驱动的校准。

## 技术栈

| 层 | 选型 | 说明 |
|----|------|------|
| 应用框架 | **Tauri 2** | Rust 主进程 + 系统 WebView,包体 ~15MB(Electron 同类约 150MB) |
| 前端 | **React 19 + TypeScript + Vite** | |
| 状态管理 | **Zustand** | 轻量,自选列表/缓存/设置 |
| 持久化 | **Tauri Store plugin** | JSON 落盘到用户配置目录 |
| 图表 | **ECharts** | 净值走势折线图 |
| 网络 | **Tauri http plugin** | 主进程发请求,绕 CORS |

## 设计要点

- **数据层不吞错**:service 抛出语义化错误类型(`NetworkError`/`FundNotFoundError`/`ParseError`),UI 据此给明确提示——吸取 fishing-funds 静默吞错(`catch { return {} }`)导致用户只看到无意义「添加失败」的教训
- **正则解析,绝不 eval**:`pingzhongdata` 是 JS 文件,用平衡括号扫描 + JSON.parse 提取,每字段独立容错
- **数据源可替换**:数据层抽象为 service,若 `pingzhongdata` 也失效可替换实现
